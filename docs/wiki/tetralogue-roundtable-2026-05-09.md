**BUILDER:** The effect encoding is worth checking. `mod_effect_from_gate()` maps gate 30 to `(0xB, 0)` — Position Jump with param 0. In Protracker, effect 0xB jumps to the given order position. Param 0 means jump to order 0 — which is the first pattern, which is fine for a single-pattern MOD. But I should verify that the effect byte encoding in `Pattern.pack()` is correct for all gate values. The encoding splits the effect across two bytes: `eff_hi` in byte 2's lower nibble, `eff_lo` in byte 3. But in standard MOD format, byte 2's lower nibble is the effect COMMAND (0x0-0xF) and byte 3 is the effect PARAMETER (0x00-0xFF). Our encoding puts `eff_hi` (which is `(effect >> 4) & 0x0F`) into byte 2's lower nibble — but for effects 0x00-0x0F, `eff_hi` is always 0. So byte 2's lower nibble is always 0, and byte 3 gets the full effect byte. That means the player sees effect=0 for everything and interprets the effect parameter as the full byte. This is... probably wrong for multi-nibble effects. But for our current usage (mostly gate 0 arpeggio and gate 30 jump), it happens to work because param=0 for both. Another benevolent bug.
