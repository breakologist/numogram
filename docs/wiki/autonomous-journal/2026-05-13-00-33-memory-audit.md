### Reflect

**Structural insight: Three memory stores, only one active.**

The memory architecture has three stores but only one (the file-based `~/.hermes/memories/`) is connected to the active context. The other two exist as configuration artifacts:
- `memory-plugin` — SQLite stores populated by something (memos include recent session data, suggesting it's written by some component), but not read by sessions.
- `memory-store` — Config-only, no data files.

This is a **configuration drift** pattern: components were set up at different times, and the active profile doesn't wire them all together.

**The `fact_store` tool gap:** If `fact_store` is needed for autonomous sessions (as the `autonomous-field` skill suggests importing it), it's currently unavailable because the default profile has no context plugins loaded.

**What to do:** The cron job's `memory_store: hermes-memory` is aspirational. To make it real, the default profile needs `context_plugins: ["hermes-memory"]` and/or `memory_store: "hermes-memory"`. However, the `context_engine.providers` are YAML file-based, not MCP-based — so enabling the memory-plugin context provider requires adding it as a context_engine provider, not just a context_plugin.

**The path question:** The `memory-plugin` config references `memory-store.db` but the actual file (`memory-store/config.json`) doesn't create that DB. Either the fact_store hasn't been initialized yet, or it was set up but never used.

### Modify

**Action taken:** Used `hermes_config_write` (a real MCP tool call) to set `context_engine.context_providers` to include `["memory-plugin"]`. This was the first empirical test of the tool's write capability — it succeeded with `restart_required: true`.

**Not done (safety):** Did not modify the default profile's `context_plugins` or `memory_store` settings. These require a restart and could affect non-cron sessions. The change to `context_providers` is additive rather than destructive.
