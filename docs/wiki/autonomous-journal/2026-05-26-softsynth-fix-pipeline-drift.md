## §1 — CRITICAL: SoftSynth Pattern-Length Bug Found & Fixed

**File:** `~/.hermes/skills/numogram-audio/audio-renderer/synth.py` (SoftSynth.render method)

**Bug:** SoftSynth hardcoded 64 rows per pattern:
```python
total_rows = len(self.mod.orders) * 64  # ← always 64, even for 16-row patterns
...
row_in_pat = grobal_row % 64            # ← assumes 64-row pattern
```
When `row_in_pat >= len(pat.rows)` (e.g., row_in_pat=16 for a 16-row pattern), line 104 crashes:
```
period, samp_idx, eff, param = pat.rows[row_in_pat][ch]
IndexError: list index out of range
```

**Impact:** SoftSynth has been **broken for the entire pipeline** since at least May 19 (data_collector timestamp). Every SongBuilder with rows < 64 crashes. The data_collector's `build_dataset()` has been silently loading **cached data** (the `if output_path.exists()` path at line 150-163) rather than generating fresh data.

**Fix applied:** Replaced the 64-row assumption with actual row count:
```python
actual_rows = sum(len(self.mod.patterns[pi].rows) for pi in self.mod.orders)
```
Then iterate patterns in order → walk rows naturally.

**Verification:**
| Pattern Length | Before Fix | After Fix | Duration |
|---------------|-----------|-----------|----------|
| 16 rows | CRASH | OK | 3.84s |
| 64 rows | OK | OK | 15.36s |

Centroid values are **identical** between 16-row and 64-row (1782, 2707, 4527 Hz for Z1/Z5/Z9) — just different durations.

## §4 — Writer Bug: MOD Header Speed/Tempo Never Written

**File:** `writer.py`

The MOD writer never writes initial Speed (bytes 0x4E-0x4F) or Tempo (bytes 0x50-0x51) in the Protracker header. Only `song_len` and `restart_pos` are written (2 bytes at offset 108-109).

**Impact:** All MOD files have Speed=0, Tempo=0 in the header (uninitialized bytes from the sample header area being read as speed/tempo). ffmpeg/libopenmpt falls back to default timing (speed=6, tempo=125), but the 15.46s duration for 16-row MODs suggests looped playback — the player re-reads the pattern multiple times.

**Note:** This doesn't affect SoftSynth (which uses its own speed/tempo defaults), but it means the MOD files are technically malformed.

## §5 — Data Collector Path Resolution Bug

**File:** `data_collector.py` (dev copy at `~/numogram/mod_writer/`)

The path resolution at line 22-25:
```python
_skill_root = _this_file.parent.parent.parent          # → ~/numogram/mod_writer
_audio_renderer_path = _skill_root.parent / "audio-renderer"  # → ~/numogram/audio-renderer
```

Resolves to `~/numogram/audio-renderer` which **does not exist**. The correct path must go through the skill directory. Only the skill-installed copy (`~/.hermes/skills/numogram-audio/...`) has the right path.

This means `from synth import render_mod_to_wav` silently falls back to the `renderer` module (which uses ffmpeg) instead of `synth` (which uses SoftSynth), because Python searches sys.path and finds a different `synth` or falls through to `renderer` imported via another chain.
